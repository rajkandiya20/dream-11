# Wallet nav overlap, admin role from Supabase, match tabs, and admin payment methods

This change addresses a cluster of user-facing issues: wallet deposit/withdrawal buttons obscured by the floating bottom nav bar, admin role not being recognized after login (because it was never fetched from Supabase), match page tabs not laying out properly horizontally, and missing admin routes. It also introduces an admin payment methods management system — admin-defined bank accounts/UPI IDs that appear in the user's deposit flow instead of hardcoded options. The approach hardcodes Supabase and Firebase credentials directly into `env.dart`, makes Supabase initialization mandatory (no longer wrapped in try/catch), and wires a `_fetchOrCreateSupabaseUser` method into both login and session-restore flows.

Watch for: Hardcoded API keys and Supabase anon key committed to source control (confirmed). Silent swallowing of all errors in admin repository CRUD methods — every operation returns `null` or `false` on failure with no diagnostics surfaced to the user (confirmed). Race condition in auth init: the user is briefly set as `role: 'user'` before the async Supabase fetch completes, which means route guards and admin UI checks can flash incorrectly during app startup (confirmed).

## High-level view

The auth fix is the most impactful change. Previously, `AuthRepository` only used Firebase data to build a `UserModel`, so the `role` field always defaulted to `'user'`. Now it queries the Supabase `users` table by UID after Firebase login succeeds, and upserts a row if none exists. The `AuthNotifier._init()` path sets a Firebase-only user model immediately (optimistic, role defaults to `'user'`) then asynchronously fetches the real Supabase profile, creating a window where `isAdmin` is false even for admins.

Credentials that were previously injected via `--dart-define` or `.env` at build time are now hardcoded string literals in `env.dart`. The Supabase anon key and Firebase API key are committed to version control. This is a security posture downgrade — the anon key grants read/write access through Supabase Row Level Security, and the Firebase API key enables unauthenticated auth operations against the project.

The admin payment methods feature spans the full clean architecture stack: repository CRUD against an `admin_payment_methods` table, provider state, a management screen with add/delete UI, and integration into the wallet deposit flow where the user-facing bottom sheet dynamically shows admin-defined methods. The wallet repository fetches active methods (filtered by `is_active: true`) in parallel with wallet load. When no admin methods are configured, the deposit sheet falls back to hardcoded UPI/Bank/PhonePe options.

The wallet bottom-padding fix is a single-line bump from 100px to 120px in the `SliverToBoxAdapter` at the end of the scroll view. The match tabs fix adds `indicatorSize: TabBarIndicatorSize.tab` and `labelPadding: EdgeInsets.symmetric(horizontal: 4)` to ensure the three tab labels distribute evenly.

This works if the column type is `jsonb`, but if it's `text`, the insert will fail silently (see issue 3). Confirm the table schema matches. (possible)
6. **No edit flow for payment methods** — `updateAdminPaymentMethod` exists in the repository and provider but the screen only exposes add and delete. Admin cannot modify an existing method's details without deleting and re-creating. (confirmed, low severity)
7. **Wallet padding fix is fragile** — The 100→120px bump assumes a specific nav bar height. If the nav bar height changes or the device has a system nav bar, the buttons may be clipped again. Derive the padding from the actual nav bar widget height or use `MediaQuery.padding.bottom`. (likely)

## Auth role fetch and the startup race

The `AuthNotifier._init()` flow is:

1. Check `currentFirebaseUser` — if non-null, immediately emit `AuthState(status: authenticated, user: UserModel(role: 'user'))`.
2. Call `_fetchSupabaseUser()` asynchronously — when it completes, emit a new state with the real role.

Between steps 1 and 2, any widget watching `isAdminProvider` sees `false`. If the router has an admin guard that checks `isAdmin` on route resolution, an admin user landing directly on `/admin` will be redirected away during this window. The fix should either (a) not emit `authenticated` until the Supabase fetch completes, or (b) add a separate `roleLoaded` flag that admin guards wait for.

The login path does not have this race — `loginWithEmail` awaits `_fetchOrCreateSupabaseUser` before returning the `AuthResult`, so the notifier receives the full user model immediately.

## Admin payment methods: silent failure pattern

Every method in the payment CRUD follows this pattern:

```dart
Future<bool> deleteAdminPaymentMethod(String id) async {
  try {
    await _client.from('admin_payment_methods').delete().eq('id', id);
    return true;
  } catch (e) {
    return false;
  }
}
```

The caller (`AdminNotifier.deletePaymentMethod`) checks the boolean but does nothing on failure — no error state is set, no snackbar is triggered, no retry is offered. From the user's perspective, tapping "Delete" and getting no feedback (the item stays in the list) is indistinguishable from the app being frozen. At minimum, set `state.errorMessage` on failure so the screen can display it.

## Supabase initialization as a hard requirement

`main()` now does:

```dart
await SupabaseClientHelper.initialize();
debugPrint('✅ Supabase initialized successfully');
```

If `Supabase.initialize()` throws (network timeout, invalid URL, SDK error), the exception propagates to `main()` and the app dies with an unhandled error. The previous code caught this and printed a warning, allowing the app to degrade gracefully. Given that Firebase is the auth backend and Supabase is the data backend, a network blip on cold start shouldn't brick the app entirely.

## Deposit flow integration with admin payment methods

The `WalletProvider` fetches admin payment methods in parallel with the wallet and transactions during `loadWallet()`:

```dart
final results = await Future.wait([
  _repository.getWallet(_userId!),
  _repository.getRecentTransactions(_userId!),
  _repository.getPaymentMethods(_userId!),
  _repository.getAdminPaymentMethods(),  // new
]);
```

The deposit bottom sheet checks `walletState.adminPaymentMethods` — if non-empty, renders those as `ChoiceChip` widgets. If empty (admin hasn't configured any, or the table doesn't exist), falls back to the hardcoded `['UPI', 'Bank', 'PhonePe']`. This graceful degradation is a good pattern. However, the selected method name is passed as the `paymentMethod` field to `initiateDeposit()` — the admin's `display_name` string. If the admin renames a method, pending transactions referencing the old name lose their association. Storing the method's `id` instead of its display name would be more robust.
