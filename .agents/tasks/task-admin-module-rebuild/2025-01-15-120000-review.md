# Admin module rebuild — full CRUD screens with image upload and cascading dropdowns

The admin module has been rewritten from six basic CRUD screens and three "manager" wrappers into six comprehensive screens with complete field forms, Supabase Storage image upload, cascading tournament-to-team-to-player dropdowns, a Playing XI selection flow, and contest auto-calculations. A new `StorageService` wraps Supabase Storage binary uploads, the repository gains four new query methods, and a SQL migration adds columns and storage buckets. Three redundant manager screens are deleted and the router updated to point directly at the rebuilt screens.

Watch for: Playing XI selection targets the wrong match ID after creation (confirmed). Silent failure throughout — every repository method and the storage service swallow exceptions returning `null`/`false`/`[]` with no logging or user-visible detail (confirmed). Storage upload path uses a timestamp but no content hash, making cache-busting unreliable and file collisions possible on concurrent uploads (likely). No RLS policies ship with the storage buckets (confirmed — SQL inserts public buckets with no accompanying policy statements).

## High-level view

The data layer adds a `StorageService` that wraps `uploadBinary` plus `getPublicUrl` and four new repository methods for cascading data (teams-by-tournament, players-by-team, match-players CRUD). These are simple Supabase calls with a universal "return empty on error" strategy that prevents the UI from ever showing error context but keeps screens from crashing.

All six screens now follow an identical local-state pattern: `ConsumerStatefulWidget` with `_loading`/`_items` managed via `setState`, data loaded in `addPostFrameCallback`, forms inside `StatefulBuilder`-wrapped `AlertDialog`s. The pattern is consistent and avoids the Riverpod build-time mutation bug that plagued the previous code. The cost is that each screen is 500-900 lines of dense widget code with no extraction of reusable form widgets or validation logic.

The Playing XI flow validates 11 players per team and at least one WK, but has a race condition: it fetches the newly-created match ID by taking `_matches.first` after a reload, which assumes ordering and relies on no concurrent match creation. This is the most likely source of a production bug.

The SQL migration is additive-only (ALTER TABLE ADD COLUMN IF NOT EXISTS, CREATE TABLE IF NOT EXISTS) and safe to run repeatedly. Storage buckets are created as public with no row-level security policies, meaning anyone with a valid Supabase anon key can upload arbitrary files to those buckets.

<details>
<summary>Issues (7)</summary>

1. **Playing XI targets wrong match** — After creating a match, `_showPlayingXIDialog` reloads all matches and assumes `.first` is the just-created one. Use the ID returned from `createMatch` instead.
2. **Silent error swallowing** — Every repository/storage method catches all exceptions and returns a default. Add at minimum a `debugPrint` or expose the error message through a return type so the admin sees what went wrong.
3. **Storage buckets have no RLS** — The SQL creates public buckets with no `storage.objects` policy. Any client with the anon key can upload to `tournament-logos`, `team-logos`, `player-photos`. Add policies restricting uploads to authenticated admins.
4. **Upload path collision** — `DateTime.now().millisecondsSinceEpoch` is used as the filename. Two concurrent uploads within the same millisecond overwrite each other. Include a UUID or hash.
5. **Hardcoded `image/png` content type** — `StorageService.uploadImage` always sets `contentType: 'image/png'` regardless of actual file type picked from gallery (could be JPEG). Detect from file extension or bytes.
6. **No input validation beyond presence** — Text fields accept unbounded input, numeric fields fall back to defaults on parse failure without telling the user. Entry fee and prize pool accept negative numbers. Add min/max constraints.
7. **`setMatchPlayers` delete-then-insert is not atomic** — If the insert fails after the delete, the match is left with zero players for that team. Wrap in a Supabase RPC or use a transaction.

## Playing XI race condition

When a match is created, `_showPlayingXIDialog` is called immediately after the create button handler pops the dialog. Inside that method, `_loadMatches()` is awaited and then `_matches.first['id']` is taken as the newly-created match ID:

```dart
await _loadMatches();
if (_matches.isNotEmpty) {
  final matchId = _matches.first['id'] as String;
  await ref.read(adminProvider.notifier).setMatchPlayers(
      matchId, selectedA.toList(), _selectedTeamAId!);
```

The match list is loaded from Supabase ordered by whatever default the query uses (the `loadMatches` call in the provider just calls the repository which does `order('date_time', ascending: false)` or similar). If two admins create matches around the same time, or the sort order doesn't guarantee the newest first, the Playing XI gets attached to the wrong match. The `createMatch` method in the repository returns a boolean, not the created record, so the ID isn't available post-creation. The fix is to change `createMatch` to return the new row's ID, then pass it directly to the Playing XI dialog.

## Silent failure pattern across the data layer

Every method in `admin_repository.dart` and `storage_service.dart` follows:

```dart
try {
  // do work
} catch (e) {
  return []; // or false, or null
}
```

This means a network timeout, a permission error, a malformed query, and a constraint violation all look identical to the UI — "nothing happened" or "empty list". For an admin panel where the operator needs to know *why* something failed (duplicate name? missing FK? storage quota exceeded?), this is particularly harmful. At minimum, log the exception; ideally, return a result type that carries the error message so the screen can show it in the snackbar.

## Storage security posture

The SQL migration inserts storage buckets with `public = true`:

```sql
INSERT INTO storage.buckets (id, name, public)
VALUES ('tournament-logos', 'tournament-logos', true)
ON CONFLICT (id) DO NOTHING;
```

No accompanying `CREATE POLICY` on `storage.objects` is defined. With Supabase's default behavior, a public bucket allows unauthenticated reads (intended for logos), but without an explicit upload policy, writes depend on the project's default storage configuration. If the default is permissive (common in development), any client can upload arbitrary content. The migration should include policies like:

```sql
CREATE POLICY "Admin upload" ON storage.objects
  FOR INSERT TO authenticated
  WITH CHECK (bucket_id IN ('tournament-logos','team-logos','player-photos')
    AND auth.jwt() ->> 'role' = 'admin');
```

## Non-atomic setMatchPlayers

`setMatchPlayers` first deletes all existing `match_players` rows for the match+team, then inserts new ones:

```dart
await _client.from('match_players').delete()
    .eq('match_id', matchId).eq('team_id', teamId);
if (playerIds.isNotEmpty) {
  await _client.from('match_players').insert(rows);
}
```

If the insert fails (constraint violation, network drop), the match is left with zero players for that team. Since the method returns `false` without context, the admin sees "could not set" but the damage (deleted rows) is already done. A Supabase RPC wrapping both operations in a transaction would make this safe.

## Hardcoded content type in StorageService

```dart
fileOptions: const FileOptions(
  upsert: true,
  contentType: 'image/png',
),
```

`image_picker` on Android/iOS frequently returns JPEG files (`.jpg`). Uploading a JPEG with `contentType: 'image/png'` means the stored object's metadata is wrong, which can break CDN caching, browser rendering in some edge cases, and any downstream image processing that trusts the content-type header. Detect the type from the file extension or use a magic-byte sniff.

## Screen code density and repetition

Each of the six screens is 500-900 lines containing the full widget tree, dialog builders, validation logic, and data loading. The `_statusBadge`, `_buildEmpty`, and form-field patterns repeat almost verbatim across screens. This isn't a blocking concern for shipping, but it's tech debt that will make the next iteration (adding pagination, search, or bulk operations) significantly harder. Extracting a `AdminCrudScreen` base or at least shared widgets for the form dialogs would reduce the surface area for bugs.
