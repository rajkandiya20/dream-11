# Home screen cleanup and fantasy team flow wiring

This change strips the home screen down to its core (live, upcoming, completed matches), fixes team logo rendering by joining the teams table in all Supabase match queries, replaces the Commentary tab with a My Team tab that shows the user's fantasy teams, and wires the captain selection screen to actually display selected players and validate before saving. The approach reuses the existing `teamBuilderProvider` family keyed by matchId but now passes the userId from `currentUserProvider` so that `saveTeam` has credentials to write.

Watch for: real-time match updates bypass the teams join and will overwrite logo data with nulls (confirmed); `debugPrint` in `_TeamLogo.build` fires twice per match card on every rebuild â€” performance concern in scrollable lists (confirmed); newly saved teams won't appear in the My Team tab without manual provider invalidation (confirmed).

## High-level view

The home repository now joins `teams!team_a_id` and `teams!team_b_id` in every match query, and `MatchModel.fromJson` cascades through nested team objects to populate flag/code fields. The real-time subscription callback in `subscribeToMatches` receives raw Postgres change payloads without joined relations, so any match update arriving over the channel will produce a `MatchModel` with null logos â€” live matches (the ones most likely to receive updates) will lose their team icons.

The `teamBuilderProvider` factory now reads `currentUserProvider` to get the userId, which means an unauthenticated user reaching captain selection will create a notifier with `userId: null` and `saveTeam` will silently return null with no user-facing explanation beyond a generic failure snackbar.

The My Team tab introduces `userTeamsForMatchProvider` as a `FutureProvider.family` keyed by a Dart record. The "Edit Team" button navigates to `/create-team/$matchId` which initializes a fresh `TeamBuilderState` â€” it doesn't pre-populate with the existing team's players, so it behaves as "Create another team" despite its label.

4. **"Edit Team" creates a new team** â€” `_FantasyTeamList`'s Edit button navigates to `/create-team/$matchId`, which starts fresh state. Existing team data is not loaded into the builder. Rename to "Create New Team" or pass team ID and pre-populate the builder.

## Real-time subscription gap for team logos

The `subscribeToMatches` method in `home_repository.dart` uses Supabase's real-time Postgres changes:

```dart
callback: (payload) {
  final newRecord = payload.newRecord;
  if (newRecord.isNotEmpty) {
    onMatchUpdate(MatchModel.fromJson(newRecord));
  }
},
```

Postgres change payloads contain the flat row from the `matches` table. There's no `team_a` or `team_b` nested object â€” those only exist when the query uses `.select()` with join syntax. So `MatchModel.fromJson` will fall through to `json['team_a_flag']` and `json['team_a_code']`, which may be null or stale depending on the table schema. Any match that gets a real-time update (score change, status transition to live) will have its logo data wiped in the UI state.

This is particularly impactful because live matches are the exact matches most likely to receive real-time updates, and they're displayed prominently at the top of the home screen.

## debugPrint in hot widget paths

`_TeamLogo` contains `debugPrint('[_TeamLogo] flag: $flag, code: $code')` at the top of its `build` method. This widget renders twice per match card (once per team), and the home screen shows up to 5 upcoming + 5 completed + any number of live matches. Every scroll-triggered rebuild prints these lines. Similarly, `_FantasyTeamCard` prints on every build. These should be removed or wrapped in a conditional that only fires once (e.g., in `initState` of a StatefulWidget, or gated behind an environment check that prevents repeated output).

## Provider cache staleness after team creation

The save flow in `_BottomBar` calls `notifier.saveTeam()`, shows a snackbar, then pops twice (back through captain-selection and create-team to match detail). At match detail, the My Team tab watches `userTeamsForMatchProvider((matchId: matchId, userId: user.uid))`. Because this is a `FutureProvider.family`, Riverpod caches the result from the first read. Without an explicit `ref.invalidate(userTeamsForMatchProvider(...))` or `ref.refresh(...)`, the tab continues showing the old (possibly empty) team list until the provider is naturally disposed and re-created.

The fix would be to invalidate the provider after a successful save, or switch to a `StateNotifierProvider` that can be mutated imperatively.

## "Edit Team" vs "Create Team" semantics

In `my_team_tab.dart`, the button displayed when teams exist says "Edit Team" but navigates to `/create-team/$matchId`:

```dart
OutlinedButton.icon(
  onPressed: () => context.push('/create-team/$matchId'),
  icon: const Icon(Icons.add),
  label: const Text('Edit Team'),
```

The `CreateTeamScreen` initializes a fresh `TeamBuilderState` (empty selected players, default team name). None of the existing team's players, captain, or name are pre-loaded. The user experience is clicking "Edit Team" and landing on an empty player selection screen.

## Captain/VC null propagation through copyWith

`TeamBuilderState.copyWith` uses the standard Dart nullable pattern:

```dart
captainId: captainId ?? this.captainId,
viceCaptainId: viceCaptainId ?? this.viceCaptainId,
```

This means passing `captainId: null` preserves the old value rather than clearing it. The `setCaptain` and `setViceCaptain` methods work around this by swapping (if new captain was VC, clear VC), but there's no path to deselect both without using the full RESET button on the create-team screen, which also clears all player selections.
