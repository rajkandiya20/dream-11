# Ball-by-ball live scoring system for admin panel (v2)

All eight issues from the v1 review have been addressed in commit 7334f27. Overs now separate display representation (`/10`) from arithmetic (`/6`). Fantasy points use incremental fetch-then-add. Undo rebuilds per-player stats from remaining ball records. Strike rotation captures the scorer's ID before swapping. RLS policies are added. Byes/leg-byes show the additional-runs dialog. The over-end popup uses `addPostFrameCallback` with a boolean guard. A `_isBusy` flag prevents concurrent scoring calls.

Watch for: RLS policies grant unrestricted write/delete to all users including anonymous, which matches the project's existing convention but means any client with the anon key can corrupt scoring data (confirmed). The undo fantasy-point recalculation only accounts for batting and bowling facets but not fielding points—a fielder who took a catch retains those 8 points even after the wicket ball is undone (confirmed). The `updateFantasyPoints` fetch-then-add is not atomic; two concurrent calls (e.g., rapid undo + score on different clients) can race (likely).

**Verdict**: APPROVED

## High-level view

The overs fix cleanly separates concerns: `totalOvers` remains the display-friendly `12.5` form for UI and storage, while a locally computed `totalLegalBalls / 6.0` drives all rate arithmetic. This is the right split—display convention preserved, math now correct.

Fantasy points shifted from absolute-set to fetch-then-add via a new `updateFantasyPoints(delta)` method. The undo path uses a complementary `setFantasyPoints(absolute)` method that recalculates totals from scratch. This creates an asymmetry: forward operations are incremental, undo is absolute. The asymmetry is deliberate and correct for undo (you need ground-truth recalculation), but the fetch-then-add on the forward path is not atomic at the database level—it reads then writes in two statements without a transaction or RPC.

The undo rewrite now iterates all remaining balls to rebuild striker, non-striker, and bowler stats from scratch, then persists the corrected scoreboard rows and recalculated fantasy points. Fielding fantasy points (catches, run outs) are not recalculated on undo—the ball record stores who fielded, but the undo logic doesn't subtract those points.

RLS is enabled with permissive policies (`USING (true)` / `WITH CHECK (true)`) matching the existing project pattern from `part2-indexes-rls-realtime.sql`. This is functionally equivalent to no restriction but follows the codebase convention, which likely assumes a service-role key for writes and anon for reads in production.

1. **Fielding fantasy points not reverted on undo** — When a wicket ball is undone, the bowler's wicket bonus and batsman stats are recalculated, but the fielder's 8 (catch) or 6 (run out) points remain in the scoreboard. Use a Supabase RPC with `points = points + $delta` to make it atomic. (likely)
3. **RLS allows anonymous write/delete** — The `ball_by_ball` policies permit any role to insert, update, and delete. If the Supabase project exposes the anon key to the client (typical), any user can manipulate ball records. This matches the project's existing pattern but is worth noting as a security posture decision. (confirmed)

## Fielding fantasy points survive undo

The undo path recalculates batsman fantasy (`runs * 1 + fours * 1 + sixes * 2`) and bowler fantasy (`wickets * 25`) from the remaining ball records, then calls `setFantasyPoints` with the absolute recalculated value. But when the original wicket ball was recorded, `updateFantasyPoints` also credited the fielder:

```dart
// In recordWicket:
if (fielderId != null) {
  if (dismissalType == 'caught') fielderPoints = FantasyPoints.catchPoints;
  else if (dismissalType == 'run_out') fielderPoints = FantasyPoints.runOut;
  await _repository.updateFantasyPoints(state.matchId!, fielderId, fielderPoints);
}
```

The undo logic never reverses this. The deleted ball's `fielder_id` and `dismissal_type` are available on the `undone` `BallEntry` object returned by `undoLastBall`, so the information needed to subtract is present—it's just not used. In practice this means undoing a caught dismissal leaves 8 phantom fantasy points on the fielder. For a fantasy cricket platform, this is a scoring integrity gap.

## Race window on fantasy point updates

The repository method:

```dart
final existing = await _client.from('scoreboard')
    .select('points').eq('match_id', matchId).eq('player_id', playerId).maybeSingle();
final currentPoints = (existing?['points'] as num?)?.toDouble() ?? 0.0;
final newPoints = currentPoints + pointsDelta;
await _client.from('scoreboard').upsert({'points': newPoints, ...});
```

Between the `select` and the `upsert`, another client (or another async call if `_isBusy` timing allows it) can perform its own read, producing a lost update. The `_isBusy` guard on the notifier eliminates same-device races for normal scoring (single user tapping), but doesn't protect against multi-device admin scenarios or the undo path which uses `setFantasyPoints` (absolute write) concurrently with a forward-path `updateFantasyPoints` on a different player. A Supabase RPC performing `UPDATE scoreboard SET points = points + $1 WHERE ...` would eliminate this class of bug.
