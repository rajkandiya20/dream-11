# Ball-by-ball live scoring system for admin panel

A complete scoring subsystem: SQL table, repository, state notifier with full ball/extra/wicket/undo logic, and a CricHeroes-style mobile-first UI with 11 new widget files. The state notifier manages overs, batsman/bowler stats, fantasy point calculation, and strike rotationâ€”all persisted to Supabase on every delivery. The UI replaces the old admin scoreboard screen entirely with a professional scoring panel.

Watch for: overs representation uses a broken encoding (`balls / 10.0` instead of `/6`) which corrupts run rate calculations and the overs display after ball 5 (confirmed). Fantasy points are overwritten rather than accumulated, so a player who bats and bowls will lose their batting fantasy points when their bowling stats update (confirmed). Undo only reverses aggregate totals but does not restore batsman/bowler individual stats or fantasy points, leaving the scoreboard inconsistent after any undo (confirmed).

## High-level view

The over-counting scheme encodes ball progress as `overNo + ballInOver / 10.0`, meaning 12 overs and 5 balls becomes 12.5, not the correct 12.833 (12 + 5/6). Every calculation that divides runs by this "overs" figure (CRR, economy, required run rate) produces progressively wrong numbers as the innings advancesâ€”and at ball 6 within an over, the representation collides with the next integer over.

Fantasy points are set as absolute values via a `points` column update, but each scoring action recomputes only one facet (batting or bowling) and overwrites the total. A batsman who later takes a wicket has their batting contribution erased.

The undo operation fetches recalculated aggregate totals from the ball table but never touches the local `striker`, `nonStriker`, or `bowler` state objects, nor the `scoreboard` rows or fantasy points already written. After undo, the UI shows correct totals but stale per-player stats.

The SQL migration creates the `ball_by_ball` table with foreign keys and indexes but no Row Level Security policies. Since the existing project runs on Supabase with client-side access, any authenticated (or anonymous) user can insert, read, or delete ball records directly.

Strike rotation on odd runs swaps `striker`/`nonStriker` in state, but the updated stats object (which just had runs/balls/fours/sixes incremented) is placed on the `nonStriker` field after swap. The next ball then attributes stats to the wrong batsman object when the cycle repeats because the `updateScoreboard` call after the swap still references `state.striker!.id`â€”which is now the *old* nonStriker who didn't score.

## Overs arithmetic is off by a factor

Throughout `scoring_provider.dart` and `scoring_repository.dart`, ball progress within an over is encoded as `overNo + (ballInOver / 10.0)`. This means after 3 overs and 4 balls the value is 3.4, not 3.667. Since run rate is computed as `totalRuns / overs`, the denominator is always too small, inflating CRR and economy as the innings progresses.

```dart
// scoring_repository.dart line ~153
final oversDouble = overs + (ballsInOver / 10.0);

// scoring_provider.dart line ~217
final newOvers = isOverComplete
    ? (overNo + 1).toDouble()
    : overNo + (_currentBallInOver / 10.0);
```

At ball 6 within an over (before `isOverComplete` triggers), the value would be `overNo + 0.6`, which visually looks fine for display ("12.6 overs") but is arithmetically wrong for rate calculations. The overs display convention in cricket *is* `12.5` for 12 overs 5 balls, but rate calculations must convert to actual overs: `12 + 5/6 = 12.833`.

The fix is to keep the display representation (`/ 10`) separate from the arithmetic representation (`/ 6`). Use the display form only in UI and storage; compute rates against `legalBalls / 6.0`.

## Fantasy points overwrite on every action

`updateFantasyPoints` writes an absolute `points` value:

```dart
await _repository.updateFantasyPoints(
    state.matchId!, state.striker!.id, fantasyPts);
```

For a batsman this recalculates `runs * 1 + fours * 1 + sixes * 2`. For a bowler on wicket, it's `wickets * 25`. If a player is an all-rounder, the bowling update (`bowlerFantasy = updatedBowler.wickets * 25`) replaces whatever batting score was stored. The fielder update sets a flat 8 or 6 rather than adding to existing totals.

This needs to either fetch the current points and add the delta, or use a Supabase RPC that performs `points = points + delta` atomically.

## Undo is aggregate-only

`undoLastBall` deletes the last row from `ball_by_ball`, refetches aggregates (total runs, wickets, overs), and updates `state`. But it never:

- Recalculates `striker.runs`, `striker.balls`, `striker.fours`, `striker.sixes`
- Recalculates `bowler.runsConceded`, `bowler.overs`, `bowler.wickets`
- Reverts `scoreboard` rows (the upsert from the now-deleted ball is still there)
- Reverts fantasy points (the absolute write persists)

After undo, the match total will be correct but the per-player scorecard will show one extra ball/run/wicket that no longer exists.

## Strike rotation identity confusion

In `recordRun`, after updating striker stats:

```dart
state = state.copyWith(
  striker: runs % 2 == 1 ? state.nonStriker : updatedStriker,
  nonStriker: runs % 2 == 1 ? updatedStriker : state.nonStriker,
  ...
);
```

If odd runs are scored, `state.striker` becomes the old `nonStriker` (unchanged stats), and `state.nonStriker` becomes `updatedStriker` (the batsman who just scored). Then the code runs:

```dart
await _repository.updateScoreboard(state.matchId!, state.striker!.id, {
  'runs': updatedStriker.runs,
  ...
});
```

`state.striker!.id` is now the *non-scoring* batsman, but `updatedStriker` holds the scoring batsman's stats. This writes the scoring batsman's stats to the wrong player's scoreboard row.

## Missing RLS on ball_by_ball

The SQL migration enables realtime but doesn't add any `CREATE POLICY` statements. Depending on the Supabase project's default (`ALTER DEFAULT PRIVILEGES`), this table is either fully open or fully locked. The other tables in this project (looking at the `sql/` directory) likely have policies; this one should match. Without it, any client with the anon key can insert or delete balls.

## Bye/leg-bye always records exactly 1 run

When the user taps "Bye" or "Leg Bye", the callback is `onExtraTapped(type, 0)`. In the notifier, `extraRuns = 1 + additionalRuns` = 1. There's no UI path for the user to record a 4-bye (ball goes to boundary as byes). The additional-runs dialog is only shown for wide/no-ball. This is a UX gap that limits scoring accuracy.
