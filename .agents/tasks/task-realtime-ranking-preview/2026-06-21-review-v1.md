# Realtime ranking pipeline and team preview redesign

This change adds a live data layer for fantasy cricket: ball-by-ball, player stats, fantasy points, and contest entries flow through Supabase realtime channels into the existing Riverpod state management. The match detail screen swaps its Commentary tab for a "My Team" tab that shows live-updating fantasy points per player with captain/VC multipliers. A companion SQL migration creates the four backing tables. The team preview screen is rebuilt as a full-screen cricket-ground layout with role-based player positioning.

**Watch for:** The `_MyTeamTab` passes an empty string as `userId` to `getUserTeamsForMatch`, which will query Supabase with `eq('user_id', '')` and always return zero results (confirmed). The `subscribeToLeaderboard` and `subscribeToContestEntries` in `LiveRankingRepository` both subscribe to the same `contest_entries` table with identical filters, creating redundant channels (confirmed). The `recalculateRanking` method issues N sequential UPDATE queries without a transaction, risking partial rank writes on failure (confirmed). RLS policies on all four new tables are wide-open (all operations allowed by anyone) — appropriate for development but a production security gap (confirmed).

## High-level view

The realtime data pipeline mirrors the existing scoreboard/commentary pattern: repository methods return a `RealtimeChannel`, callbacks parse `payload.newRecord`, and the notifier merges updates into immutable state lists. The new channels for player stats and ball-by-ball integrate cleanly into `MatchDetailNotifier` with proper dispose. The contest-level realtime layer introduces a `LiveRankingRepository` that duplicates its own subscription logic — the leaderboard and contest-entries channels filter on the same table and column, differing only in channel name and callback shape.

The "My Team" tab's `FutureBuilder` fetches teams on every build with a hardcoded empty `userId`, meaning it will never show data until authentication integration lands. This makes the tab a dead end at runtime despite the UI being structurally complete.

The SQL migration defines `contest_entries.fantasy_team_id` as `NOT NULL`, but the Dart model treats it as nullable (`String?`), so `toJson()` conditionally omits the field — an insert from Dart would violate the database constraint if `fantasyTeamId` is null.

The team preview redesign is a standalone visual change with no data-layer coupling beyond accepting an optional `playerPoints` map. It groups players by role using `Wrap` and applies captain/VC multipliers for display.

## Duplicate realtime channels on contest_entries

`LiveRankingRepository` exposes `subscribeToLeaderboard` and `subscribeToContestEntries`. Both open a Postgres changes subscription on the `contest_entries` table, filtered by `contest_id`:

```dart
// subscribeToLeaderboard
.onPostgresChanges(
  table: 'contest_entries',
  filter: PostgresChangeFilter(column: 'contest_id', value: contestId),
  ...
)

// subscribeToContestEntries — identical table and filter
.onPostgresChanges(
  table: 'contest_entries',
  filter: PostgresChangeFilter(column: 'contest_id', value: contestId),
  ...
)
```

The only difference is the channel name and callback type (`ContestEntryModel` in both cases). In `ContestDetailNotifier._subscribeToRealtime()`, both are subscribed simultaneously. The leaderboard callback does in-memory re-sort; the contest-entries callback triggers a full `getLeaderboard()` network fetch. For every single row change on `contest_entries`, both callbacks fire — the second one (network fetch) will overwrite whatever the first one (optimistic re-sort) just set in state, producing a visible flicker as the state changes twice in rapid succession.

The likely intent was for `subscribeToLeaderboard` to watch the existing `leaderboard` table (which exists per `part1-drop-and-create-tables.sql`) and `subscribeToContestEntries` to watch `contest_entries`. The current implementation conflates them.

## Empty userId making My Team tab non-functional

```dart
// match_detail_screen.dart, _MyTeamTab
future: fantasyRepository.getUserTeamsForMatch(
  userId: '', // Will use authenticated user
  matchId: matchId,
),
```

The comment acknowledges this is a placeholder but the code ships with it. `FantasyRepository.getUserTeamsForMatch` applies `.eq('user_id', userId)` with the empty string, which matches no rows. The tab renders its empty state ("No team created yet") for every user regardless of whether they have teams.

Additionally, this `FutureBuilder` is inside a `ConsumerWidget.build()` with no caching. Every time the parent `TabBarView` rebuilds (tab switch, state change), a new future is created, firing another Supabase request. Converting to a `FutureProvider.family` keyed on `(userId, matchId)` would deduplicate the requests and cache results.

## recalculateRanking sequential writes without transaction

```dart
for (final entry in ranked) {
  await _client
      .from('contest_entries')
      .update({'rank': entry.rank})
      .eq('id', entry.id);
}
```

For a contest with 100 entries this issues 100 sequential HTTP requests. If the 50th fails, entries 1-49 have updated ranks and 50-100 don't. Supabase supports RPC calls — a PostgreSQL function that sorts and updates in a single transaction would be both faster and atomic. Alternatively, `supabase-dart` supports batch operations via `.upsert()` with a list.

The method also silently swallows errors and returns an empty list, making it impossible for the caller to distinguish "no entries" from "update failed."

## SQL schema vs Dart model nullability

The SQL migration defines:
```sql
CREATE TABLE contest_entries (
  ...
  fantasy_team_id UUID NOT NULL REFERENCES fantasy_teams(id) ON DELETE CASCADE,
  ...
);
```

But `ContestEntryModel` declares:
```dart
final String? fantasyTeamId;
```

And `toJson()` conditionally includes it:
```dart
if (fantasyTeamId != null) 'fantasy_team_id': fantasyTeamId,
```

If this model is ever used for an INSERT and `fantasyTeamId` happens to be null, Postgres will reject the row with a NOT NULL violation. The Dart type should be non-nullable (`String`) to match the schema and catch this at compile time.

The same asymmetry exists for `user_id` on `contest_entries` — SQL declares `TEXT NOT NULL` while the Dart model uses a required non-nullable `String`, which is correct. The inconsistency is only on `fantasyTeamId`.

## RLS policies: all tables wide open

All four new tables get identical RLS policies:
```sql
CREATE POLICY "read_all" ON ball_by_ball FOR SELECT USING (true);
CREATE POLICY "write_all" ON ball_by_ball FOR INSERT WITH CHECK (true);
CREATE POLICY "update_all" ON ball_by_ball FOR UPDATE USING (true);
CREATE POLICY "delete_all" ON ball_by_ball FOR DELETE USING (true);
```

This means any authenticated (or anonymous, depending on Supabase config) client can insert arbitrary ball-by-ball records, modify fantasy points, or delete contest entries. For a development/prototype phase this is standard, but `contest_entries` and `fantasy_points` contain ranking and prize data that should be write-restricted to service-role or specific RPC triggers before any real users are on the system.

## Realtime publication references leaderboard table

The migration adds to the publication:
```sql
ALTER PUBLICATION supabase_realtime ADD TABLE leaderboard;
ALTER PUBLICATION supabase_realtime ADD TABLE fantasy_teams;
ALTER PUBLICATION supabase_realtime ADD TABLE fantasy_team_players;
```

The `leaderboard` table is defined in `part1-drop-and-create-tables.sql` (pre-existing), so this is valid. However, `fantasy_teams` and `fantasy_team_players` are also pre-existing tables being added to the publication here — if those tables were already in the publication from a previous migration, this statement will error on re-run. This is not idempotent. Consider wrapping in `DO $$ ... IF NOT EXISTS` guards or accepting that migrations only run once.
